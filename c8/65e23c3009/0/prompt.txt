# Plan: Simplify `entire enable` Onboarding Flow

## Context

The current `entire enable` flow is verbose and implementation-focused (mentions strategies, individual hook types, directory creation). The goal is to make it clean and user-friendly:

- **First run**: Shows agent, installs everything, asks telemetry, says "Ready."
- **Second run**: Detects already-enabled state and shows a brief summary instead of re-running setup.

## Target Output

**First run:**
```
entire enable
Agent: Claude Code (use --agent to change)

✓ Hooks installed
✓ Project configured (.entire/settings.json)

  Help improve Entire CLI?
  Share anonymous usage data. No code or personal info collected.
  [Yes] [No]

Ready.
```

**Second run (already fully enabled):**
```
entire enable
Already enabled. Everything looks good.

  Agent: Claude Code
  Config: .entire/settings.json
```

## Changes

### File: `cmd/entire/cli/setup.go`

#### 1. Add `isFullyEnabled()` helper function

Checks whether Entire is already fully set up:
- Settings exist and `Enabled == true`
- Agent hooks installed (`checkClaudeCodeHooksInstalled()` — already exists)
- Git hooks installed (`strategy.IsGitHookInstalled()` — already exists)
- `.entire` directory exists (`checkEntireDirExists()` — already exists)

Returns `(isEnabled bool, agentDescription string, configPath string)`.

#### 2. Rewrite `runEnableInteractive()`

New flow:
1. Check `isFullyEnabled()` — if true, print the "already enabled" summary and return early
2. Print `Agent: Claude Code (use --agent to change)` (detect agent via `agent.Default()`)
3. Print blank line
4. Install Claude Code hooks + git hooks → print `✓ Hooks installed` (single combined line)
5. Setup `.entire` directory + save settings → print `✓ Project configured (.entire/settings.json)` (or `settings.local.json`)
6. Print blank line
7. Ask telemetry consent (existing `promptTelemetryConsent`)
8. Print blank line + `Ready.`

Key simplifications:
- No mention of strategy in output (still uses default internally)
- Combined "hooks installed" message (agent hooks + git hooks)
- Combined "project configured" message (directory + settings)
- No settings target prompt on first run (first run always creates `settings.json`)
- End with "Ready." instead of strategy name

#### 3. Update `runEnableWithStrategy()` to match

Apply similar output simplifications for the `--strategy` flag path — same structure but skip "already enabled" check since the user is explicitly requesting a strategy change.

#### 4. Update `setupAgentHooksNonInteractive()` output

When `--agent` flag is used, keep it as non-interactive but update messages to be consistent.
